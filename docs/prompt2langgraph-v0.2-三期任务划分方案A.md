# prompt2langgraph v0.2 三期任务划分方案A

## 1. 文档目的

本文档用于明确 `prompt2langgraph` v0.2 的三期任务划分方案，作为后续正式任务计划与实施排期的上位输入。

本文档仅给出 **分期目标、阶段范围、交付重点与版本边界**，**暂不展开详细实施步骤、文件级修改清单或逐项开发计划**。

---

## 2. 分期策略说明

v0.2 采用 **方案A：目标链路优先**。

该方案的核心思路是：优先补齐项目距离目标描述最关键的缺口，先打通 `Prompt / Plan → WorkflowSpec → 可执行 LangGraph` 的最小闭环，再逐步增强真实执行能力与 Skill 转换能力。

相较于“真实执行器优先”或“工程底座优先”的方案，方案A更适合作为 v0.2 的主线，原因如下：

1. 更贴合项目目标定位，即从用户输入到 LangGraph 工作流的完整链路。
2. 能在 v0.2 阶段形成更完整、可演示、可验证的产品闭环。
3. 能在不一次性扩张范围的前提下，为真实 LLM 执行器、Tool 执行器和 Skill 转换建立稳定入口。

---

## 3. v0.2 总体目标

v0.2 的总体目标是：

- 在现有 v0.1 的 IR 校验、编译、运行框架基础上，补齐输入层与关键执行层能力；
- 让系统从“只能处理结构化 Workflow IR / 简化 JSON plan”的状态，演进为“可以从 Prompt 或受约束文本计划出发，生成并执行工作流”的状态；
- 为后续 Skill 到可执行工作流、Join 控制流补齐、生产级持久化等能力奠定基础。

---

## 4. 三期任务划分

## 第一期：输入适配闭环（Input Bridge）

### 4.1 阶段目标

打通 `Prompt / Text Plan / JSON Plan → WorkflowSpec` 的最小闭环，使用户不必直接编写完整 Workflow IR，也能够进入现有的校验、编译、运行流程。

### 4.2 主要任务

1. **文本计划适配器正式化**
   - 梳理现有 `text_plan` 相关实现的真实状态与边界。
   - 明确受支持的 `plan_text` 输入格式。
   - 将文本计划稳定转换为规范 `WorkflowSpec`，并复用现有验证与编译运行链路。

2. **Prompt → JSON Plan 生成入口**
   - 设计并实现基于 LLM 的计划生成入口。
   - 先生成简化 JSON plan，再进入现有适配器与校验流程。
   - 对 LLM 输出增加 JSON 解析、结构校验与诊断输出，确保不能绕过现有验证体系直接执行。

3. **CLI / Public API 输入扩展**
   - 为文本计划或 Prompt 输入增加清晰的 CLI 入口。
   - 对外暴露对应的 Public API，保持现有接口行为兼容。

4. **诊断与测试补齐**
   - 补充非法 JSON、字段缺失、未知节点、类型不匹配等输入错误的诊断场景。
   - 增加文本计划 / Prompt 相关 fixtures 与 CLI / API 回归测试。

### 4.3 阶段交付重点

- 用户可从 Prompt 或受约束文本计划出发，得到可校验的 `WorkflowSpec`。
- 生成结果能够继续进入 `validate / compile / run / graph` 流程。
- 输入链路完成后，项目从“仅结构化输入”升级为“支持上层输入适配”的编译运行系统。

### 4.4 本阶段暂不追求

- 不要求 Skill 已能生成可执行工作流。
- 不要求所有 `llm` 节点已接入真实模型执行。
- 不要求复杂自然语言自由解析达到高泛化能力。

---

## 第二期：真实执行能力补齐（Real Execution）

### 5.1 阶段目标

在已打通输入闭环的基础上，引入真实 LLM 执行能力与受控 Tool 执行能力，使工作流从 mock 运行演进为具备真实业务执行潜力的系统。

### 5.2 主要任务

1. **LLM Provider 抽象**
   - 定义 provider 配置、模型标识、采样参数等结构。
   - 建立统一的 LLM 调用抽象层，为不同模型来源预留接入接口。

2. **真实 LLM Executor**
   - 在保留现有 mock executor 的前提下，引入真实 LLM executor。
   - 支持从工作流状态中组织 prompt/messages，并返回结构化运行结果。
   - 运行时记录必要的 provider、model、耗时、token 摘要等观测信息，但避免泄露 secret。

3. **Tool Executor 最小安全模型**
   - 为 `tool` 节点补齐最小可用执行器。
   - 初期仅支持受控注册工具或受限函数调用，不开放任意 shell 或任意外网能力。
   - 将参数校验、输出约束纳入现有 registry / typecheck 框架。

4. **策略与安全约束增强**
   - 为真实外部调用引入显式启用策略。
   - 默认保持安全与可控，避免隐式外部调用。
   - 为后续 `side_effect` 节点执行预留审批与幂等能力接入点。

5. **集成测试策略补齐**
   - 单元测试继续使用 fake provider 或 mock provider。
   - 真实外部依赖测试改为显式启用的集成测试。
   - 确保 v0.1 已有 deterministic 行为不被新能力破坏。

### 5.3 阶段交付重点

- `llm` 节点可以在 mock 模式之外选择真实 provider 执行。
- `tool` 节点具备最小安全可用执行能力。
- 项目能够在保证默认安全的前提下，支撑更接近真实场景的工作流运行。

### 5.4 本阶段暂不追求

- 不追求一次性覆盖所有模型供应商或高级调用模式。
- 不追求开放式、无限制的工具执行能力。
- 不强制在本阶段完整交付 `side_effect` 执行闭环。

---

## 第三期：Skill 与控制流补全（Skill & Control Flow Completion）

### 6.1 阶段目标

把 Skill 分析能力推进到可生成工作流的方向，并补齐 `join` 控制流与 `side_effect` 最小闭环，提升系统对目标能力的覆盖度与完整性。

### 6.2 主要任务

1. **Skill → WorkflowSpec 转换器**
   - 基于现有 `SkillDirectoryAnalysis` 与 `draft_nodes` 结果，设计 Skill 到工作流节点的映射规则。
   - 将可识别的 Skill 步骤转换为可校验、可编译的 `WorkflowSpec`。
   - 对风险资源或高风险语义保留诊断和人工审批边界。

2. **Skill 参数注入与资源建模**
   - 支持从 CLI / API 向 Skill 工作流注入参数。
   - 明确 scripts / assets / references 在工作流中的表示方式。
   - 保持“分析资源而非默认执行资源”的安全边界。

3. **Join 边执行支持**
   - 补齐 `join` 边在编译器和运行时中的执行语义。
   - 明确分支合并条件、聚合策略与 reducer 约束。
   - 同步更新验证、Mermaid 表达、fixtures 与回归测试。

4. **Side Effect 最小执行闭环**
   - 在二期安全模型基础上，为 `side_effect` 节点提供带审批或幂等键约束的最小执行器。
   - 保持外部副作用执行的显式授权与可审计性。

5. **运行时状态边界增强**
   - 优化当前对 `InMemorySaver` 内部结构的耦合。
   - 抽象更清晰的持久化边界，为后续生产级 checkpointer 做准备。

### 6.3 阶段交付重点

- Skill 不再仅限静态分析，而是具备进入工作流编译链路的能力。
- `join` 控制流从“仅可表达”升级为“可执行”。
- `side_effect` 节点具备最小安全闭环。
- 运行时状态管理边界更加明确，为后续演进降低风险。

### 6.4 本阶段暂不追求

- 不要求完整生产级数据库持久化方案全部落地。
- 不要求生成完全自包含、脱离运行时库的静态代码包。
- 不将 Web UI、服务化、性能并行优化纳入 v0.2 主体范围。

---

## 5. v0.2 建议版本边界

### 5.1 建议纳入 v0.2 的核心范围

建议 v0.2 至少完成以下内容：

- 第一期全部内容；
- 第二期中的真实 LLM executor 核心能力；
- 与上述能力对应的诊断、测试与 CLI / API 支撑。

### 5.2 可作为 v0.2 增强项的内容

以下内容可视开发进度作为 v0.2 增强项：

- `tool` 节点最小安全执行能力；
- Skill → WorkflowSpec 的 alpha 版本；
- `join` 边执行支持；
- `side_effect` 节点最小闭环。

### 5.3 不建议纳入 v0.2 的内容

以下内容不建议纳入 v0.2 主版本目标：

- Web UI / HTTP 服务化；
- 多目标平台编译；
- 完全自包含静态代码生成；
- 完整生产级数据库持久化；
- 任意 shell 或不受控外部工具执行能力。

---

## 6. 推荐实施顺序

建议按以下顺序推进：

1. **先完成输入适配闭环**，确保项目真正向“Prompt → LangGraph”目标迈出关键一步；
2. **再补齐真实执行能力**，让工作流具备真实场景可用性；
3. **最后补 Skill 与控制流能力**，提升系统完整性与扩展性。

这种顺序有利于在 v0.2 周期内持续形成可验证成果，并降低多条主线并行开发带来的复杂度与风险。

---

## 7. 下一步建议

在本三期划分确认后，下一步再进入详细计划阶段，按每一期分别拆解：

- 目标范围；
- 模块改动点；
- 测试补充范围；
- 阶段验收标准；
- 实施优先级与依赖关系。

当前阶段先以本分期文档作为 v0.2 任务规划基线，不展开详细实施计划。
